## 1. サイトの目的定義

**一言**：ヒューマノイド導入を検討するtoB事業者が、**ここだけ見れば現場実装で気になる変数を体系的に理解できる**、日本語の入口メディア。

- 人型ロボット限定。
- ECサイトのような分かりやすいUIで「探す→理解する」を最短にする。
- カタログ等のコンテンツは保有が前提（table stakes）。差別化は**UXと体系性**、および後発の一次情報。
- 媒体本体は無料。収益は将来の実務側（選定支援・PoC設計・レンタル仲介等）。MVPでは売らない。

**このサイトが"名刺"として一瞬で伝えるべきこと**：何を作ろうとしている人間か／本気度と質／連絡先。網羅性ではない。

---

## 2. 想定読者の整理

- **主**：導入を検討し始めたtoB（製造・物流・小売・介護・施設運営など）の**知識ゼロ層**。役割は現場責任者・新規事業・DX/設備担当。
- **副（MVP時点で実在）**：カンファレンスで名刺交換した相手（メーカー・代理店・SIer・同業）。→ Aboutと旗艦記事で「本気度」を判定される。
- **将来**：toC（市場成熟と余力次第で転用）。MVPでは無視。

---

## 3. 読者の課題と導線

読者の理解が進む順 ＝ サイトの順路（「言いたいこと順」ではなく「納得する順」）。

| ステージ | 読者の問い | 当てる場所 |
|---|---|---|
| 知る | 何ができて、何ができない？うちに本当に要る？ | 導入ガイド（基礎・期待値調整） |
| 探す | どの機種がある？うちの作業に合うのは？ | ロボットカタログ＋用途から探す |
| 判断する | コスト・安全・設置・保守・調達はどうなる？ | 導入ガイド（意思決定変数）★中核 |
| 動く | で、どう進める？誰に相談する？ | 導入ガイド（導入プロセス）＋相談 |

★「判断する」＝このサイトの存在意義。競合（ヒューマノイドプレス/Metoree等）が構造的に持たない層。

## 4. 推奨グローバルナビ

フル構成の実装ナビ：

1. `ロボット` … 機種カタログ。導入判断軸で探す。
2. `メーカー` … 供給体制DB。メーカー、代理店、SIer、サポート体制を見る。
3. `比較` … 候補機種を導入判断変数で比較する。
4. `ガイド` … 知る／判断する／動く をまとめた理解ハブ。
5. `用途から探す` … 業界ではなく、タスク・作業起点で候補を探す。
6. `記事` … reports。市場動向、導入レポート、政策、取材、分析。
7. `会社概要` … ミッション、運営者、信頼材料。
8. `お問い合わせ` … 情報提供、掲載相談、取材相談、導入相談。

公開時は中身が薄いページを非表示にしてよい。ただし実装上のURLとデータモデルはフル構成で持つ。

## 5. トップページのセクション構成

Homeは「サイトの入口」ではなく、全体の使い方を30秒で理解させるハブにする。

1. **ヒーロー**：日本企業向けのヒューマノイド導入判断ポータルであることを一文で伝える。
2. **主要導線**：ロボット、メーカー、比較、ガイド、用途から探す、記事を6カードで出す。
3. **判断変数の提示**：価格、調達、国内可否、保守、安全、PoC、用途適合をこのサイトの軸として見せる。
4. **注目ロボット/メーカー**：公開情報で埋められるものだけを少数表示。
5. **用途から探す導線**：倉庫ピッキング、工場点検、受付案内などタスク起点の入口を出す。
6. **最新reports**：ニュースではなく「買い手にとってなぜ重要か」が分かる記事を見せる。
7. **運営者/問い合わせ**：情報提供、掲載相談、取材相談、導入相談につなげる。

## 7. 各ページの役割

- **Home**：30秒で「何のサイトで・誰向けで・本気で・連絡先はここ」を伝える名刺。
- **About**：信頼の中核。ミッション、なぜ作るか、運営者、今後のロードマップ（透明性）。名刺交換相手はここで判断する。
- **導入ガイド（記事）**：知識ゼロを"理解した状態"に運ぶ。MVPの旗艦は「意思決定変数の地図」＝サイトの存在意義そのもの。
- **ロボットレコード**：機種を導入判断レンズで記述（スペック＋調達・コスト・安全・設置・国内可否）。EC的詳細ページ。
- **メーカー**：会社紹介ではなく、日本での販売・代理店・サポート・PoC体制を見る供給体制ページ。
- **比較**：カードを並べるだけでなく、導入判断変数の比較表を主役にする。
- **用途から探す**：業界紹介ではなく、作業・タスク起点で成立条件と候補ロボットを探す。
- **記事・レポート（reports）**：一次情報で鮮度と信頼を供給。大物取材＝補強、ハンズオン/失敗事例＝堀。
- **相談（contact）**：ゆるい副CTA。MVPはメール/SNS DM＋簡易フォームで十分。

## 8. 各ページに書くべき内容

**Home**：① コンセプト1文 ② 想定読者1行 ③ 解決する3不安 ④ 旗艦記事リンク ⑤ 透明性（今後の計画）⑥ 運営者一言 ⑦ CTA。

**About**：① ミッション（なぜこの入口が必要か）② 課題認識（既存情報は買い手目線で体系化されてない）③ 提供価値（体系性＋一次情報）④ 運営者プロフィール（経歴・なぜ自分か）⑤ ロードマップ（カタログ→用途逆引き→一次取材→相談/支援）⑥ 連絡先。

**旗艦ガイド「意思決定変数の地図」**：導入を判断する前に押さえる変数を体系化。
- 調達形態（買い切り/リース/RaaS/サブスク）
- 総保有コスト（本体＋周辺＋保守年額＋消耗バッテリー＋運用人件＋ティーチング工数）
- 安全・法規制（労働安全、電波法/技適、人協働の評価）
- 設置・運用要件（床/段差、充電インフラ、ネットワーク、防塵防水等級）
- 運用体制（誰が面倒を見る／必要スキル）
- ROI/投資回収の考え方
- ベンダー継続性リスク（黎明期は"会社が消える"が現実の変数）
- 最後に「ヒューマノイドじゃなくていいケース」を正直に書く＝信頼を取りにいく。

**ロボットレコード**：導入判断変数（下記 §9 の `Robot` 型フィールド）を埋める。出典必須。

**相談**：何を相談できる/できないかを明記（今は商品はない＝期待値調整）。更新通知導線を併設。

## 9. データ型定義（Next.js / TypeScript）

`data/types.ts` に集約する。各データファイル（`data/robots.ts` 等）はこの型を import して使う。

真実源は `nextjs_data_types_v1.ts`。Next.jsプロジェクト作成後は、その内容を `data/types.ts` にコピーする。

- `Manufacturer` … メーカー・代理店・SIer・AI OS企業まで拡張できる供給体制カタログ
- `Robot` … 機種カタログ（中核データ）。`manufacturerId` で Manufacturer を参照
- `Guide` … 理解コンテンツ（learn / evaluate / act）
- `UseCase` … 用途・活用シーン。業界そのものではなく、タスク起点の逆引き導線
- `Report` … 一次情報・記事・レポート

> services / works は**意図的に未定義**。最初から空の型を作らない。

## 10. 型定義（data/types.ts）

ここには型を二重管理しない。`nextjs_data_types_v1.ts` を参照する。

重要な実装ルール：

- 公開URLの識別子は `slug`
- 旧UIの `id` は `slug` に変換する
- `posts` ではなく `reports`
- `industries` ではなく `useCases`
- 出典は `string[]` ではなく `Source[]`
- enum値は英語トークンにし、UI側で日本語ラベルに変換する
