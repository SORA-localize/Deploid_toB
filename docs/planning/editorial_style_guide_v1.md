# 記事編集スタイルガイド v1

Last reviewed: 2026-07-01

> このガイドは、Deploid に掲載する記事（`data/articles.ts`）の執筆・編集方針を定める。
> データフィールドの仕様は `data/types.ts`、公開前チェックは `data-maintenance-checklist-v1.md` C項を参照。

## 1. 基本方針

### 対象読者

日本のtoB事業者における導入検討者・意思決定者。技術に詳しくないが、現場と予算を持っている人を想定する。「面白い」ではなく「自社に置き換えて考えられる」を基準に書く。

### 立ち位置

- 転載・要約サイトではなく、公開情報を構造化・解釈する編集メディアである
- プレスリリースの翻訳にならない。「何が起きたか」より「なぜ重要か・何を意味するか」に字数を使う
- 根拠のある数字と出典を必ず持つ。AIの推測や曖昧な一般論は書かない

### 文体

- **だ・である調**で統一する
- 一人称・呼びかけは使わない（「〜だろう」は可、「〜しましょう」は不可）
- 見出しは体言止めか短い文。長い見出しは避ける
- 正確さを保ちながら、冒頭は「読者の判断が動く差分」から入る。注意書きや免責から始めすぎると報告書になる
- 筆者の視点は、主観的な感想ではなく「導入検討者ならここで引っかかる」という編集判断として出す
- 読点で短い節を積まない。助詞・接続助詞で繋げる箇所を読点で切らず（「〜は、〜で、〜し、〜た。」を避ける）、読点は一文に2つまでを目安にする
- GIGAZINE型の「具体的な差分・数字・新しさを先に出す」構造と、Gizmodo型の「読者が自分ごと化できる違和感・体験・問いから入る」構造を参考にする。ただし一人称・過度な軽さ・煽りは使わない

### 出典名の扱い

本文では、媒体名をむやみに主語にしない。出典は `sources[]` に残るため、本文の役割は出典名の列挙ではなく、事実と意味の整理である。

原則:

- 「Wiredによれば」「The Robot Reportは」「ロボスタの記事では」のような媒体名主語を避ける
- 通常は「報道では」「公式発表では」「論文では」「発表資料では」程度に抽象化する
- 文末や段落ごとに出典名を差し込まない
- 出典名を本文に出す場合は、読者の判断に必要なときだけにする

出してよい例:

- Nature など、学術論文であること自体が信頼性・段階判断に関係する場合
- 政府文書、規制当局文書、官報、標準化団体文書
- 公式発表主体そのものが論点の場合（メーカー公式、導入企業公式、IRなど）
- 調査会社レポートや独占報道で、誰の調査・誰の報道かが数字の意味に直結する場合

悪い例:

```markdown
Wiredによれば、NEOの手は25自由度を備える。
The Robot Reportは、遠隔手術の可能性を報じている。
```

良い例:

```markdown
NEOの手は25自由度を備えると報じられている。
今回の論文では、生体を対象にした前臨床の実現可能性試験として位置づけられている。
```

## 2. NG表現リスト

以下の語・表現はAI生成特有の癖として禁止する。類似表現も同様に避ける。

**修飾語・評価語**
- 「革新的な」「画期的な」「前例のない」「注目に値する」「見逃せない」
- 「大きな一歩」「歴史的な」「転換点となる」「マイルストーン」（多用禁止）
- 「まさに」「まさしく」「いよいよ」

**強調・誇張**
- 「急速に」「急拡大」「ますます加速」（数字と根拠なく使う場合）
- 「業界全体を変える」「パラダイムシフト」「ゲームチェンジャー」
- 「世界初」「業界初」（出典なく使う場合）

**まとめ・接続詞**
- 「まとめると」「総じて言えば」「結論として」（文章が構造化されていれば不要）
- 「〜として注目される」「〜に期待が集まる」
- 「引き続き注視が必要だ」「今後の動向が注目される」（これだけで終わる段落）

**その他**
- 一段落に体言止めを3回以上連続させる
- 箇条書きを多用して文章の論理をごまかす（セクション別ルールを参照）

## 3. 記事執筆ワークフロー

1つの記事を書く前に、以下の順で判断する。

### Step -1: 記事とシリーズの設計

記事は単発の情報ではなく、読者の調査を前に進めるための部品として扱う。

- `棚` はサイト上の大分類
- `シリーズ` は同じ観点で揃えた記事の束
- `記事` は1つの問いに答える単位

メーカー解説・ロボット解説は、主要プレイヤーを初めて把握したい読者向けの常設シリーズとして作る。ニュースのように「今なぜ見るか」を主題にせず、同じ観点で比較できる粒度を揃える。

### Step 0: 棚と記事種別の判断

`/reports` の公開棚は `すべて / ニュース / メーカー解説 / ロボット解説 / 基礎知識` とする。実装上の正本は `lib/articleShelves.ts`。

| 棚 | `type` の扱い | 判断基準 |
|---|---|---|
| ニュース | 既存のニュース系 `ArticleType` | 新しく起きた外部事実で、業界理解または導入判断の変数が動くもの |
| メーカー解説 | `manufacturer-guide` | 主要メーカーの位置づけ、製品群、強み・弱み、導入/実証状況、日本での調達導線を整理する記事 |
| ロボット解説 | `robot-guide` | ロボット詳細ページとは別に、その機体の重要性、向く用途、導入制約、比較軸を読む記事 |
| 基礎知識 | `basics-guide` | 時間軸に依存しにくい、ヒューマノイド導入・業界理解の基本を整理する記事 |

ニュースで扱うもの:

- 実証・商用導入・契約
- 新モデル・価格・販売条件・RaaS
- 量産・資金調達・IPO・M&A・提携
- 政策・規制・安全基準・補助金
- 大手企業の導入準備・実装に近い提携

ニュースで扱わないもの:

- バズ動画だけ
- 論文単体
- 出典1本の噂
- 中身の薄いプレスリリース
- 「AI搭載」「世界初」だけで、何が変わるか不明な発表

`メーカー解説` と `ロボット解説` は、既存のメーカー/ロボット詳細ページを本文化したものにしない。詳細ページはデータベース、解説記事は「導入検討者がどう読むべきか」を説明する読み物として役割を分ける。

### Step 1: ネタの収集と一次確認

- 公式プレスリリース、メーカー公式サイト、信頼できる業界媒体（Nikkei Asia、The Robot Report、Digitimes等）を確認する
- 数字・固有名詞・日付を一次出典で裏取りする。裏取りできないものは書かない
- `sources[].url` にHTTPアクセスして生きていることを確認する（404/403のまま公開しない）
- 出典が1件しか取れない場合はドラフトのまま置く
- ニュース記事はサムネイルなしで公開しないことを原則にする。実機画像が使えない場合は、権利確認済みの汎用テーマ画像を当てる
- 画像は本文の事実根拠ではなく、記事テーマを示す表示素材として扱う。実機ではない汎用画像を使う場合は `permissionNote` にその旨を書く

### Step 2: セクション判断

以下の6セクションのどれに当てはまるかを決める。判断軸は「この記事で読者が得る情報の種類」。

| セクション | `section` 値 | 判断基準 |
|---|---|---|
| ニュースダイジェスト | `digest` | 複数トピックを意図的に1本の記事にまとめる特別記事。自動の日次記事候補では原則使わない |
| 導入・事例 | `deployment` | 実際のロボット導入・運用・契約の話が中心か |
| 市場・動向 | `business` | 資金調達・IPO・業界数値・企業戦略の話か |
| 技術・製品 | `tech` | 新製品・新機能・研究成果の話か |
| 政策・規制 | `policy` | 政府方針・法規制・補助金・国家戦略の話か |
| 話題・その他 | `entertainment` | 上記に分類しにくい、社会的話題・バイラル・イベントか |

複数にまたがる場合は「記事の核心が何か」で決める。例：資金調達→ `business`、その資金で作った新製品→ `tech`。自動の日次収集では、複数ニュースを束ねず、記事化できるニュースを個別記事として分ける。

### Step 3: セクションの型を適用して執筆

ニュース系の記事は次のセクション別テンプレートに沿って `body` を書く。`manufacturer-guide` / `robot-guide` / `basics-guide` は §6 の解説系テンプレートを使う。見出し（h2）は型のラベルをそのまま使わず、内容に即した自然な言葉にする。

### Step 4: 最終監査（公開前に必ず実行する）

原稿が完成したら、公開・完了報告の前に本文全体へ次の監査を通す。1つでも直したら再度通す。

1. **事実監査（ハルシネーション検出）**：本文中のすべての数字・固有名詞・日付・断定を `sources[]` の該当箇所と突合する。出典で確認できない記述は削除するか「確認できる公開情報はない」に書き換える。出典が言っていないことへの言い換え・拡大解釈（例：「〜台販売」→「世界1位」）も同様に落とす
2. **誇大表現監査**：§2 NG表現リストと照合する。「世界トップクラス」「〜を誇る」「〜をリードする」「他を圧倒する」のような出典なしの優位断定も同類として落とす
3. **日本語の自然さ監査**：通読（音読相当）して、読点で節を積んだ細切れ文、「や」「の」の連続、話題の宣言し直し、助詞の誤用、だ・である調の乱れを直す。並列の列挙は「・」で揃える
4. **重複監査**：一つの事実・一つの数字が二箇所以上に出ていないか確認する（正本のセクション以外から削る。§6-1a ルール3）

## 4. 共通構造

どのセクションでも、記事本文は以下の4つの役割を果たす段落で構成する。

| 役割 | 文量の目安 | やること |
|---|---|---|
| 冒頭（リードとして機能） | 150〜300字 | 核心を先出し。「誰が・何を・した/決めた」を最初の1〜2文で言い切る |
| 事実の整理 | 300〜600字 | 数字・固有名詞・時系列をソースから正確に展開する |
| 背景・文脈 | 200〜400字 | なぜ今これが起きているか。構造的な理由を書く |
| バイヤー示唆 | 200〜400字 | 日本のtoB事業者として何を考えるべきか。抽象論ではなく具体的な問いを立てる |

**`whyItMatters` との役割分担**

- `whyItMatters`：導入検討者向けの核心まとめ。1〜2文のエレベーターピッチ。`body` を読まなくても意味が取れるように書く
- `body`：上記4役割を展開した本体。`whyItMatters` の繰り返しにしない

**見出し（h2）の書き方**

「リード」「事実の整理」「バイヤー示唆」などのラベルをそのまま見出しにしない。各段落が何を言っているかを短く表す独自の表現にする。見出しを読んだだけで記事の骨格が分かる状態が理想。

```
悪い例:
## 背景
## まとめ

良い例:
## Fremont 工場で何が起きているか
## 1,000台超のGen3、COGS目標は2万ドル台
## 日本のサプライヤーが見ておくポイント
```

## 5. セクション別テンプレート

### 5-1. 導入・事例（deployment）

**目的**: 読者が「自社・自業種に置き換えて考えられる」情報を提供する。「どこで・誰が・何台・何をさせているか」の事実を軸にする。

**核心で答える問い**
- 誰が、どの現場で、何台使っているか
- なぜそのロボット・メーカーを選んだか
- 実際の運用条件・制約は何か
- 同種の現場を持つ日本企業は何を確認すべきか

**典型的な構成例**

```markdown
## （現場・導入規模を端的に示す見出し）
核心を先出し。「X社がY工場にZを100台導入した」という事実から始める。

## （選定理由・条件の見出し）
なぜこのロボットか、なぜこの現場か。稼働条件・前提。

## （実運用の現実を示す見出し）
出てきた課題・成果・数値。プレスリリースにない情報があれば優先して書く。

## （読者の現場に引きつける見出し）
この事例から日本のtoB事業者が参照できるポイント。「〜の現場を持つ企業は〜を検討する材料になる」など具体的に。
```

**注意**
- 「初の導入事例」「業界初」は出典なく書かない
- 台数・場所・契約形態は必ず出典を確認する
- 「成功した」と断言しない。発表段階か、稼働中か、効果測定済みかを区別する

---

### 5-2. 市場・動向（business）

**目的**: 読者が「今がどういう時期か」「何に投資・注目すべきか」を判断できる材料を渡す。感想ではなく数字と構造で語る。

**核心で答える問い**
- 何の数字が、どう動いたか
- なぜ今この動きが出ているか（規制・需要・技術成熟のどれか）
- この動きは一過性か、構造的変化か
- 日本の事業者にとって何を意味するか

**典型的な構成例**

```markdown
## （数字・動向を端的に示す見出し）
核心の数字または動向を先出し。「X社が〇億ドルを調達」「市場規模がX年比Y倍に」など。

## （数字の意味を示す見出し）
その数字はどこから来たか。他社・昨年比・市場全体との対比で文脈を与える。

## （背景・構造的要因の見出し）
なぜ今この数字が出ているか。市場・規制・技術の構造を書く。

## （日本市場・日本企業への含意の見出し）
この動向が日本のtoB事業者の判断にどう影響するか。
```

**注意**
- 「急拡大」「急成長」は数字と根拠セットで使う
- 資金調達額・IPO評価額は公式発表か信頼できる一次報道を出典にする
- 予測数字は「〇〇社調べ」「レポートによれば」と明記し、予測と事実を混ぜない

---

### 5-3. 技術・製品（tech）

**目的**: 「この技術が何を可能にして、何をまだできないか」を整理する。バズワードより仕様と制約を重視する。

**核心で答える問い**
- 何ができるようになったか（具体的な動作・数値）
- 既存技術・既存製品との差分は何か
- 実用上の制約・前提条件は何か
- どんな現場・用途に使えそうか、使えないか

**典型的な構成例**

```markdown
## （技術・製品の核心を示す見出し）
何ができるのかを先出し。「〇〇が△△を実現した」の形で。

## （仕様・動作の実態を示す見出し）
発表・論文・デモから取れる具体的な数字・条件・制約。

## （既存との違いを示す見出し）
類似製品・前世代・競合との比較。「初めて」は出典なく使わない。

## （実用化の距離感を示す見出し）
研究段階か、製品化済みか、量産はいつか。現場に入るまでのギャップを正直に書く。
```

**注意**
- 「業界を変える技術」系の評価は書かない。事実と数字で読者に判断させる
- デモ映像の成功を「実用化」と言い換えない
- 論文ベースの場合はピアレビュー済みか、プレプリントかを区別する

---

### 5-4. 政策・規制（policy）

**目的**: 日本のtoB事業者が「何を・いつまでに知っておくべきか」を整理する。法律・補助金・国家戦略の読み解きは具体性で勝負する。

**核心で答える問い**
- 何が決まったか・決まろうとしているか
- いつから・誰に適用されるか
- 日本企業が対応・参照すべき内容は何か
- 情報源はどこか（官報・省庁サイト・法律番号など）

**典型的な構成例**

```markdown
## （決定事項の核心を示す見出し）
何が決まったかを先出し。「〇省が〜を決定した」「〜法が改正された」の形で。

## （内容の詳細を示す見出し）
対象範囲・金額・スケジュール・条件を整理する。

## （背景・意図を示す見出し）
なぜこの施策が出てきたか。政策的文脈。

## （日本企業の対応ポイントを示す見出し）
この政策を受けてtoB事業者が確認すべきこと。「〜の補助金を申請できる事業者は〜」など具体的に。
```

**注意**
- 「注視が必要だ」だけで終わらない。具体的な対応・確認先を示す
- 施行日・適用範囲・対象者は必ず出典（官報・省庁）を確認する
- 日本以外の政策を扱う場合は「日本への影響」を必ず書く

---

### 5-5. 話題・その他（entertainment）

**目的**: 業界の空気・社会的文脈を伝えながら、toB的な含意を必ず引き出す。「面白かった」で終わらない。

**核心で答える問い**
- なぜDeploidがこれを取り上げるか（toB視点での理由）
- 事象の事実は何か
- 業界・市場にとって何を示唆するか

**典型的な構成例**

```markdown
## （事象を端的に示す見出し）
何が起きたかを先出し。

## （事実の詳細を示す見出し）
起きたことの詳細。数字・固有名詞・経緯。

## （業界的含意を示す見出し）
なぜこれがヒューマノイド業界のトレンドを示すか。必ずtoB視点の解釈を入れる。
```

**注意**
- バイラル動画・展示デモは「実用化」と書かない
- `whyItMatters` に「toB的な理由」を必ず書く。含意が書けないネタは掲載しない

## 6. 解説系テンプレート

### 6-1. メーカー解説（manufacturer-guide）

**目的**: ヒューマノイド業界をこれから理解する読者が、主要メーカーごとの違いを把握できるようにする。渡すべきは「その会社の詳しい知識」ではなく「その会社を導入検討上どう扱えばいいか」。読み終えたときに、その会社を一言で分類できる状態（例：「低価格・量産寄りの中国メーカー。商用実績と地政学リスクは要確認。研究・教育・PoCの初期候補として強い」）を目指す。

**分量**: 本文合計 4,000〜6,000字を標準とする（実績リスト・ラインナップ表・チャネルリストの構造情報は別）。検索経由の読者は1ページで全容（いくらで・何が買えて・どこに聞けばいいか）を求めるため、「詳細は他ページへ」で本文を痩せさせない。ロボット詳細ページとの重複はスペックの丸写しを避けることで防ぎ、価格・位置づけの要点は本記事内でも完結させる。

**見出しは固定テンプレートを使う**

メーカー解説は同じ観点で複数社を比較できることが価値なので、§4の「見出しラベルをそのまま使わない」という一般則の例外として、次の6本の見出しをどのメーカーでも同じ順・同じ語で使う：**会社概要 → 製品ラインナップ → 沿革 → 導入実績 → 日本からの調達 → よくある質問**。見出し文字列と順序の正本はコード側（`lib/manufacturerGuideTemplate.ts`）で、本文データはセクション別の固定テンプレート型（`data/types.ts` の `ManufacturerGuideContent`）に入れる。

見出しを揃えるだけでは記事は揃わない。**各セクションは、順序固定の役割（骨格）に分解してある**。執筆は「骨格の各枠に、出典で確認できた情報を書き込む → 枠の順序のまま自然な日本語で接続する」の2段階で行う。役割名・枠の番号は本文に出さない。骨格に対応する情報が確認できない枠は、推測で埋めず「確認できる公開情報はない」と地の文で明記する。

**枠を接続するときの禁止事項**（枠埋め方式はここで崩れやすい）：

- 枠の切り替わりで「〜の特徴は」「〜の点では」のように話題を宣言し直さない。前の枠から続けて読める文で始める
- 読点で短い節を積まない。「人型への転換は、〜として進んだ。」のように助詞で繋げる箇所を読点で切らず、「人型への転換は〜として進み、」と一文に編む。読点は一文に2つまでを目安にする
- 全枠を埋めたら通読し、細切れのリズムと話題の宣言し直しを潰してから完成とする

### 会社概要（目安400〜600字）

4つの役割を、この順序のまま自然な文章としてつなげる。役割名・番号は本文に出さない。

| 役割 | 目安字数 | 必ず含める要素 |
|---|---|---|
| ①基本識別 | 100〜150字 | 正式名称（現地語表記があれば併記）、創業年、本社所在地。この3点は省略しない |
| ②事業構造 | 100〜150字 | 事業の柱の数と関係（カテゴリ単位。個々の機種名・価格帯は書かない＝製品ラインナップ節に譲る）。ヒューマノイドがこの会社の本業か、新規事業・実験的位置づけかを明記する。売上規模やヒューマノイド出荷比率が確認できる場合は数字を入れる |
| ③差別化要因 | 100〜150字 | 価格・性能・供給力のどれで競争しているか一つ選び、それを支える構造的理由（内製化、垂直統合、資本力、提携網など）を書く。根拠のない「安い」「高性能」は書かない |
| ④象徴情報 | 100〜150字 | 下の優先順位リストに従い、一次情報で確認できた最初の指標だけを一つ使う |

**④象徴情報の優先順位**（上から順に判定し、最初に確認できた項目を採用する。複数該当しても1つに絞る）：

1. 出荷実績・生産台数（直近年の具体的な数字）
2. 上場区分・時価総額（上場企業の場合）
3. 資金調達額・直近ラウンドの評価額（未上場企業の場合）
4. 主要顧客・提携先の実名（大手との契約が公表されている場合）
5. 特許数・研究開発投資額
6. 受賞歴・標準化団体での役職（1〜5がいずれも確認できない場合の最終手段）

該当する数字・事実が一つも確認できない場合、この役割自体を省略せず「規模を示す公開情報は確認できない」と地の文で明記する。存在しないことにして書かない。

**沿革・直近ニュース（IPO審査通過のような時点性のある事実）はここに入れない**。ニュース要約に見えたら失敗。

### 製品ラインナップ（目安300〜500字＋カード＋表）

3つの部品を、この順序で構成する。

| 部品 | 形式 | 必ず含める要素 |
|---|---|---|
| ①全体紹介文 | 文章（100〜200字） | シリーズがいくつあり、それぞれが何向けか（用途・価格帯での束ね方）。スペック順で並べない。人型以外の製品ライン（四足・部品など）があれば1文の事実として列挙する（「本記事では人型に絞る」のようなメタ言及は書かない） |
| ②値付けの説明 | 文章（150〜300字） | なぜこの価格帯構成なのか（値付け戦略・原価構造）。「安い」で終わらせない。価格の前提（税・送料・保守は含まない等）を1文添える。①から続けて読める書き出しにする |
| ③モデルカード＋解説表 | UI部品（`lineup`） | ロボットカードの横スクロールと、機体×位置づけ×価格目安の表がコード側で描画される |

`lineup` の行は `roleLabel`（どの検討者に関係するか一言）と `priceLabel`（出典で確認済みの価格目安。ロボット詳細の priceNote と矛盾させない。未確認は「要確認」）だけ書く。機体名・リンク・画像はDB（`robotId`）から自動解決される。詳細スペックの羅列はロボット詳細ページに譲る。

### 沿革（目安500〜800字）

3フェーズ固定。各フェーズは「時期＋事業の変化＋裏付け数字1つ以上」で構成し、この順で自然な文章としてつなげる。フェーズ名・番号は本文に出さない。

| フェーズ | 内容 | 裏付け数字の例 |
|---|---|---|
| ①創業〜事業基盤の確立 | 何から始まり、最初に何で立ったか | 初期製品の価格、調達額、初期顧客数 |
| ②現在の主力事業への転換 | ヒューマノイド事業がどう立ち上がったか | 発売価格、価格改定幅、出荷開始数 |
| ③直近1〜2年の拡大 | 今どの局面にいるか | 出荷台数、売上、受注額、上場・調達 |

製品名は代表例として触れる程度にとどめる。「製品発売年表」になっていたら失敗。会社概要と同じ数字を使い回さない（1つの数字は1つのセクションが正本）。

### 導入実績（リード文＋固定5分類のリスト・時系列順）

リード文1文＋固定のリストで構成する。リード文はリストを導入するだけの文にする（例:「公開情報で確認できる実装段階を、研究から商用まで5分類で整理する」）。業界内での位置づけや他メーカーとの比較はここに書かない（論拠を示す場所がない）。リストの後に要約文も置かない。

表示は囲い枠を持たない簡素なリストとし、各分類に内容を示すアイコン（研究=GraduationCap等）を付け、確認状態はモノクロの濃淡で表す（confirmed=チェックマーク付き強調／limited=通常／none=グレーアウト。新しい色は持ち込まない）。行の根拠リンクは「参照:」で示す（`design_system_v1.md` §本文ブロック方針に従う）。ヒューマノイドの商用実績はまだ業界全体で薄く、このセクションは記事の主役ではない。

分類は実装段階の時系列順に固定する（分類名・順序の正本は `lib/labels.ts` / `lib/display.ts`）：

1. 研究・教育：大学・研究機関での利用
2. 展示・デモ：イベント、動画、PR
3. PoC：顧客名・現場・目的はあるが、本番運用か不明なもの
4. 社内実証：自社工場や自社施設でのテスト
5. 商用導入：第三者企業の現場で、業務として使われているもの

各行は次の3変数で書く（ステータス文言は持たない）：

- `evidence`：視覚表示の正本。`confirmed`（確認済み＝チェックマーク付き強調）／`limited`（部分的・間接的な情報のみ＝通常表示）／`none`（公開情報なし＝グレーアウト）
- `body`：その段階について何が確認できたか（導入・実演・報道・自社発表・公開情報なし）を直接書き、顧客名・台数・件数・金額が確認できるものは数字で書く。「あり／限定的」のような判定語だけの文にしない。現場の詳細描写はロボット解説に譲る
- `sourceUrls`（`evidence` が `none` 以外は必須）：根拠URL。記事の `sources[]` に登録したURLと一致させる

読者が知りたいのは「世界全体で見て、このメーカーはどの段階まで到達しているか」である。日本での事例が確認できる場合は、地域を別軸にせず、該当する段階の行の根拠・本文中の事実として示す。

### 日本からの調達（目安150〜400字＋チャネルリスト）

文章だけで構成することは禁止。次の2部で構成する。

| 部品 | 形式 | 内容 |
|---|---|---|
| ①チャネルリスト | 構造化データ（`procurementChannels`） | 窓口を全部挙げる。各行は `kind`（公式販売／国内代理店／導入支援）＋名前＋URL＋役割一言（販売のみか、導入支援・保守まで含むか）。Deploidの問い合わせ窓口はコード側で末尾に自動追加される |
| ②固有リスク | 文章 | **このメーカーに固有の**調達リスク・注意点だけを書く（輸出規制・制裁リスト・国内サポート不在・契約形態の特殊性など）。リスク段落は「※調達リスクとして、」で書き始める（「調達リスクとしては、」にしない）。どのメーカーにも当てはまる一般的な確認項目（税・送料・保証・保守・部品納期などの羅列）は書かない——読者ごとに同じ文章が量産されるだけで情報価値がなく、基礎知識記事に譲る |

国内窓口が確認できない場合は、チャネルリストに公式問い合わせ窓口だけを載せ、その旨を文章で明記する。確認できない項目は「要確認」と書き、推測で補わない。記事公開後も代理店の追加・変更を追跡し、古くなったら更新する。

### よくある質問（該当がある場合のみ）

FAQは本文の再掲欄ではない。次の3条件のどれかに該当する場合だけ書き、該当がなければ `faq: []` とする（セクションとFAQPage構造化データごと非表示になる）：

1. 型番・呼称の混同が実際に起きている（型落ち機と後継機、旧社名・旧ブランド名など）
2. 販売終了・仕様変更など、本文の流れに組み込むと沿革・ラインナップを壊す単発の事実確認
3. リサーチで実際に見つかった読者の疑問で、他5セクションのどこにも収まらないもの（想像で作らない）

他5セクションを書き終えた後に判定する。先に本数を決めてブレストしない。答えに本文にない事実を使う場合は `sources[]` に出典を追加する。

**注意**
- メーカー詳細ページの項目を順に説明するだけの記事にしない
- 公式サイトの会社紹介を言い換えただけにしない
- 「主要メーカーだから扱う」前提で書き、ニュース的な「今見る理由」を無理に作らない
- 数字（売上・出荷台数・価格・件数）を使えるところで使う。定性評価だけの記事は薄くなる
- 供給力、価格、導入実績を断定する場合は、公式発表・IR・信頼できる報道を出典にする
- 日本対応が不明な場合は「不明」「要確認」と書き、推測で補わない

### 6-1a. メーカー解説の執筆優先ルール

この節は、上記の文量目安・各セクションの詳細より優先する。

1. **記事ごとに主張を一つ決める。** 例：`低価格で実機を入手しやすいが、商用現場の裏付けは別に見るメーカー`。全セクションは、この主張に必要な根拠だけを前へ進める。
2. **段落は「読者の疑問 → 答え → 根拠」の順に書く。** 集めたソースの順、年表の順、SEOキーワードの順に情報を並べない。
3. **一つの事実は一度だけ書く。** 価格はラインナップ表、段階別の実績は導入実績表、窓口は日本からの調達のチャネルリストを正本にする。表が答えた内容を直後の地の文やFAQで要約し直さない。
4. **導入実績は評価ではなく、根拠の種類を示す。** 各行の `body` の中で何が確認できたか（導入・実演・報道・自社発表・公開情報なし）を直接書き、`あり / 限定的 / 要確認` のような判定語だけで済ませない。各行に `sourceUrls` で根拠URLを付ける。
5. **事実は直接書く。** 「公開情報では」「確認が必要」「〜と見るべき」「〜したい」を段落ごとに繰り返さない。出典・日付・分類で正確性を担保し、未公表の事実は一度だけ「公表されていない」と書く。
6. **関係のない話題を接続詞でつながない。** 沿革には事業の変化だけを書く。規制・制裁リスト・地政学は、その情報が調達条件に影響する場合に限り、日本からの調達セクションへ置く（上場・調達は沿革③の裏付け数字として扱える）。
7. **本文で他メーカーを比較対象として挙げたら、`relatedManufacturerIds` に追加する。** 単なる出典社・投資家・技術提携先は、比較・調達判断の対象でない限り追加しない。

完成原稿では、読み手が同じ結論を本文・表・FAQで二度以上読まないことを必ず確認する。

### 6-2. ロボット解説（robot-guide）

**目的**: ヒューマノイド業界をこれから理解する読者が、主要機体ごとの違いを把握できるようにする。「どんなロボットか」「何が得意で何が苦手か」「どこで使われているか」「日本からどう接点を持てるか」を整理する。

**核心で答える問い**
- どんなロボットか（形状、サイズ、カテゴリ、メーカー内での位置づけ）
- 何が強みで、何が弱み・制約か（動作、ペイロード、自由度、安定性、価格、運用）
- どこに導入済み、またはどんな実証中か（顧客、現場、作業、台数、商用/PoCの区別）
- 日本からの購入・相談導線はあるか（代理店、国内パートナー、問い合わせ先、保証・保守の確認点）
- どの用途に向き、どの用途にはまだ早いか

**典型的な構成例**

```markdown
## （どんなロボットかを端的に示す見出し）
形状、サイズ、カテゴリ、メーカー内での役割を説明する。

## （主要仕様と能力の見出し）
自由度、ペイロード、稼働時間、可動範囲、制御方式など、性能の骨格を出典ベースで整理する。

## （強みと弱みの見出し）
運動性能、安定性、量産性、価格、ソフトウェア、保守、安全などを、確認済み事実と未確認点に分けて書く。

## （導入・実証の状況を示す見出し）
どの顧客・現場・作業で使われているか、商用導入かPoCか、台数や期間が確認できるかを整理する。

## （日本からの調達・相談導線を示す見出し）
代理店、国内パートナー、公式問い合わせ、保証、保守、ソフトウェア更新、契約形態の確認点を書く。

## （どの用途に向くかを示す見出し）
この機体が候補になる現場と、まだ早い現場を分けて説明する。
```

**注意**
- ロボット詳細ページのスペック表を文章化しただけの記事にしない
- デモ映像の成功を実運用能力として扱わない
- 「主要機体だから扱う」前提で書き、ニュース的な「今見る理由」を無理に作らない
- 「安い」「高性能」などの評価語は、比較対象と出典がある場合だけ使う
- 価格や販売条件は税、送料、保証、構成差を分けて書く
- 商用導入、PoC、展示デモ、社内利用は区別する
- 向く用途だけでなく、向かない用途も書く

### 6-3. 基礎知識（basics-guide）

**目的**: ニュースや個別機体の前提になる、時間軸に依存しにくい基本概念を整理する。

**扱う内容**
- ヒューマノイドとは何か、何ができるか
- 導入検討の一般的な手順・注意点
- PoC、遠隔支援、安全、保守、データ利用などの基本論点
- 業界構造・市場全体の概観

**注意**
- 薄い用語集や一般論を量産しない
- 個別企業・個別機体の話に寄りすぎる場合は、メーカー解説またはロボット解説として扱う
- `basics-guide` 棚はコンテンツが揃うまで UI 上では disabled とする

## 8. CLAUDE.md との関係

このガイドは `CLAUDE.md` の「デザイン方針」と同じ位置づけのコンテンツ方針として扱う。

- **コードを書く前に参照**：記事データを追加・編集する前にこのガイドを確認する
- **AI生成本文の禁止**：本文（`body`）を AIに一発生成させてそのまま載せない。調査・構造・出典確認を人が行い、NG表現を除去した上で掲載する
- **スキーマとの整合**：`data/types.ts` の `ArticleType` / `ArticleSection` / `ArticleCategory` と本ガイドのセクション分類は対応している。型を間違えると validate が落ちる
