# 記事編集スタイルガイド v1

Last reviewed: 2026-06-15

> このガイドは、Deploid に掲載する記事（`data/articles.ts`）の執筆・編集方針を定める。
> データフィールドの仕様は `data/types.ts`、公開前チェックは `data-maintenance-checklist-v1.md` C項を参照。

---

## 1. 基本方針

### 対象読者

日本のtoB事業者における導入検討者・意思決定者。技術に詳しくないが、現場と予算を持っている人を想定する。「面白い」ではなく「自社に置き換えて考えられる」を基準に書く。

### 立ち位置

- 転載・要約サイトではなく、公開情報を構造化・解釈する編集メディアである
- プレスリリースの翻訳にならない。「何が起きたか」より「なぜ重要か・何を意味するか」に字数を使う
- 根拠のある数字と出典を必ず持つ。AIの推測や曖昧な一般論は書かない

### 文体

- **だ・である調**で統一する
- 一人称・呼びかけは使わない（「〜だろう」は可、「〜しましょう」は不可）
- 見出しは体言止めか短い文。長い見出しは避ける

---

## 2. NG表現リスト

以下の語・表現はAI生成特有の癖として禁止する。類似表現も同様に避ける。

**修飾語・評価語**
- 「革新的な」「画期的な」「前例のない」「注目に値する」「見逃せない」
- 「大きな一歩」「歴史的な」「転換点となる」「マイルストーン」（多用禁止）
- 「まさに」「まさしく」「いよいよ」

**強調・誇張**
- 「急速に」「急拡大」「ますます加速」（数字と根拠なく使う場合）
- 「業界全体を変える」「パラダイムシフト」「ゲームチェンジャー」
- 「世界初」「業界初」（出典なく使う場合）

**まとめ・接続詞**
- 「まとめると」「総じて言えば」「結論として」（文章が構造化されていれば不要）
- 「〜として注目される」「〜に期待が集まる」
- 「引き続き注視が必要だ」「今後の動向が注目される」（これだけで終わる段落）

**その他**
- 一段落に体言止めを3回以上連続させる
- 箇条書きを多用して文章の論理をごまかす（セクション別ルールを参照）

## 3. 記事執筆ワークフロー

1つの記事を書く前に、以下の順で判断する。

### Step 1: ネタの収集と一次確認

- 公式プレスリリース、メーカー公式サイト、信頼できる業界媒体（Nikkei Asia、The Robot Report、Digitimes等）を確認する
- 数字・固有名詞・日付を一次出典で裏取りする。裏取りできないものは書かない
- `sources[].url` にHTTPアクセスして生きていることを確認する（404/403のまま公開しない）
- 出典が1件しか取れない場合はドラフトのまま置く

### Step 2: セクション判断

以下の5セクションのどれに当てはまるかを決める。判断軸は「この記事で読者が得る情報の種類」。

| セクション | `section` 値 | 判断基準 |
|---|---|---|
| 導入・事例 | `deployment` | 実際のロボット導入・運用・契約の話が中心か |
| 市場・動向 | `business` | 資金調達・IPO・業界数値・企業戦略の話か |
| 技術・製品 | `tech` | 新製品・新機能・研究成果の話か |
| 政策・規制 | `policy` | 政府方針・法規制・補助金・国家戦略の話か |
| 話題・その他 | `entertainment` | 上記に分類しにくい、社会的話題・バイラル・イベントか |

複数にまたがる場合は「記事の核心が何か」で決める。例：資金調達→ `business`、その資金で作った新製品→ `tech`。

### Step 3: セクションの型を適用して執筆

次のセクション別テンプレートに沿って `body` を書く。見出し（h2）は型のラベルをそのまま使わず、内容に即した自然な言葉にする。

---

## 4. 共通構造

どのセクションでも、記事本文は以下の4つの役割を果たす段落で構成する。

| 役割 | 文量の目安 | やること |
|---|---|---|
| 冒頭（リードとして機能） | 150〜300字 | 核心を先出し。「誰が・何を・した/決めた」を最初の1〜2文で言い切る |
| 事実の整理 | 300〜600字 | 数字・固有名詞・時系列をソースから正確に展開する |
| 背景・文脈 | 200〜400字 | なぜ今これが起きているか。構造的な理由を書く |
| バイヤー示唆 | 200〜400字 | 日本のtoB事業者として何を考えるべきか。抽象論ではなく具体的な問いを立てる |

**`whyItMatters` との役割分担**

- `whyItMatters`：導入検討者向けの核心まとめ。1〜2文のエレベーターピッチ。`body` を読まなくても意味が取れるように書く
- `body`：上記4役割を展開した本体。`whyItMatters` の繰り返しにしない

**見出し（h2）の書き方**

「リード」「事実の整理」「バイヤー示唆」などのラベルをそのまま見出しにしない。各段落が何を言っているかを短く表す独自の表現にする。見出しを読んだだけで記事の骨格が分かる状態が理想。

```
悪い例:
## 背景
## まとめ

良い例:
## Fremont 工場で何が起きているか
## 1,000台超のGen3、COGS目標は2万ドル台
## 日本のサプライヤーが見ておくポイント
```

## 5. セクション別テンプレート

### 5-1. 導入・事例（deployment）

**目的**: 読者が「自社・自業種に置き換えて考えられる」情報を提供する。「どこで・誰が・何台・何をさせているか」の事実を軸にする。

**核心で答える問い**
- 誰が、どの現場で、何台使っているか
- なぜそのロボット・メーカーを選んだか
- 実際の運用条件・制約は何か
- 同種の現場を持つ日本企業は何を確認すべきか

**典型的な構成例**

```markdown
## （現場・導入規模を端的に示す見出し）
核心を先出し。「X社がY工場にZを100台導入した」という事実から始める。

## （選定理由・条件の見出し）
なぜこのロボットか、なぜこの現場か。稼働条件・前提。

## （実運用の現実を示す見出し）
出てきた課題・成果・数値。プレスリリースにない情報があれば優先して書く。

## （読者の現場に引きつける見出し）
この事例から日本のtoB事業者が参照できるポイント。「〜の現場を持つ企業は〜を検討する材料になる」など具体的に。
```

**注意**
- 「初の導入事例」「業界初」は出典なく書かない
- 台数・場所・契約形態は必ず出典を確認する
- 「成功した」と断言しない。発表段階か、稼働中か、効果測定済みかを区別する

---

### 5-2. 市場・動向（business）

**目的**: 読者が「今がどういう時期か」「何に投資・注目すべきか」を判断できる材料を渡す。感想ではなく数字と構造で語る。

**核心で答える問い**
- 何の数字が、どう動いたか
- なぜ今この動きが出ているか（規制・需要・技術成熟のどれか）
- この動きは一過性か、構造的変化か
- 日本の事業者にとって何を意味するか

**典型的な構成例**

```markdown
## （数字・動向を端的に示す見出し）
核心の数字または動向を先出し。「X社が〇億ドルを調達」「市場規模がX年比Y倍に」など。

## （数字の意味を示す見出し）
その数字はどこから来たか。他社・昨年比・市場全体との対比で文脈を与える。

## （背景・構造的要因の見出し）
なぜ今この数字が出ているか。市場・規制・技術の構造を書く。

## （日本市場・日本企業への含意の見出し）
この動向が日本のtoB事業者の判断にどう影響するか。
```

**注意**
- 「急拡大」「急成長」は数字と根拠セットで使う
- 資金調達額・IPO評価額は公式発表か信頼できる一次報道を出典にする
- 予測数字は「〇〇社調べ」「レポートによれば」と明記し、予測と事実を混ぜない

---

### 5-3. 技術・製品（tech）

**目的**: 「この技術が何を可能にして、何をまだできないか」を整理する。バズワードより仕様と制約を重視する。

**核心で答える問い**
- 何ができるようになったか（具体的な動作・数値）
- 既存技術・既存製品との差分は何か
- 実用上の制約・前提条件は何か
- どんな現場・用途に使えそうか、使えないか

**典型的な構成例**

```markdown
## （技術・製品の核心を示す見出し）
何ができるのかを先出し。「〇〇が△△を実現した」の形で。

## （仕様・動作の実態を示す見出し）
発表・論文・デモから取れる具体的な数字・条件・制約。

## （既存との違いを示す見出し）
類似製品・前世代・競合との比較。「初めて」は出典なく使わない。

## （実用化の距離感を示す見出し）
研究段階か、製品化済みか、量産はいつか。現場に入るまでのギャップを正直に書く。
```

**注意**
- 「業界を変える技術」系の評価は書かない。事実と数字で読者に判断させる
- デモ映像の成功を「実用化」と言い換えない
- 論文ベースの場合はピアレビュー済みか、プレプリントかを区別する

---

### 5-4. 政策・規制（policy）

**目的**: 日本のtoB事業者が「何を・いつまでに知っておくべきか」を整理する。法律・補助金・国家戦略の読み解きは具体性で勝負する。

**核心で答える問い**
- 何が決まったか・決まろうとしているか
- いつから・誰に適用されるか
- 日本企業が対応・参照すべき内容は何か
- 情報源はどこか（官報・省庁サイト・法律番号など）

**典型的な構成例**

```markdown
## （決定事項の核心を示す見出し）
何が決まったかを先出し。「〇省が〜を決定した」「〜法が改正された」の形で。

## （内容の詳細を示す見出し）
対象範囲・金額・スケジュール・条件を整理する。

## （背景・意図を示す見出し）
なぜこの施策が出てきたか。政策的文脈。

## （日本企業の対応ポイントを示す見出し）
この政策を受けてtoB事業者が確認すべきこと。「〜の補助金を申請できる事業者は〜」など具体的に。
```

**注意**
- 「注視が必要だ」だけで終わらない。具体的な対応・確認先を示す
- 施行日・適用範囲・対象者は必ず出典（官報・省庁）を確認する
- 日本以外の政策を扱う場合は「日本への影響」を必ず書く

---

### 5-5. 話題・その他（entertainment）

**目的**: 業界の空気・社会的文脈を伝えながら、toB的な含意を必ず引き出す。「面白かった」で終わらない。

**核心で答える問い**
- なぜDeploidがこれを取り上げるか（toB視点での理由）
- 事象の事実は何か
- 業界・市場にとって何を示唆するか

**典型的な構成例**

```markdown
## （事象を端的に示す見出し）
何が起きたかを先出し。

## （事実の詳細を示す見出し）
起きたことの詳細。数字・固有名詞・経緯。

## （業界的含意を示す見出し）
なぜこれがヒューマノイド業界のトレンドを示すか。必ずtoB視点の解釈を入れる。
```

**注意**
- バイラル動画・展示デモは「実用化」と書かない
- `whyItMatters` に「toB的な理由」を必ず書く。含意が書けないネタは掲載しない

## 6. 文量・品質の基準

| `type` | 最低文量 | 目安 |
|---|---|---|
| `news-brief` | 800文字 | 核心と示唆を圧縮した短報 |
| `market-analysis` `analysis` `policy-update` | 1,500文字 | 数字・背景・含意を展開する |
| `deployment-report` `case-study` | 1,500文字 | 事実の詳細と現場示唆を展開する |
| `tech-update` | 1,200文字 | 仕様・制約・実用距離感を整理する |
| `event-report` | 1,000文字 | 場の要点と業界示唆をまとめる |
| `interview` | 2,000文字以上 | 対話構造と独自情報を含む |

## 7. CLAUDE.md との関係

このガイドは `CLAUDE.md` の「デザイン方針」と同じ位置づけのコンテンツ方針として扱う。

- **コードを書く前に参照**：記事データを追加・編集する前にこのガイドを確認する
- **AI生成本文の禁止**：本文（`body`）を AIに一発生成させてそのまま載せない。調査・構造・出典確認を人が行い、NG表現を除去した上で掲載する
- **スキーマとの整合**：`data/types.ts` の `ArticleType` / `ArticleSection` / `ArticleCategory` と本ガイドのセクション分類は対応している。型を間違えると validate が落ちる
