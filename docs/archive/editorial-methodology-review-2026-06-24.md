# Deploid 記事方法論レビュー草案

Status: 承認・統合済み（2026-07-20）
Created: 2026-06-24

> **統合済み注記**: この草案は承認され、`../decisions/editorial_style_guide_v1.md` §1.5「題材選定基準」へ統合済み。以後は同ファイルが正本。本書は経緯・草案時点の記録として残す。

この文書は、Deploid の記事を増やす前に、題材選定・切り口・書き方の基準を揃えるためのレビュー用草案である。承認前のため、現行の正本ではない。承認後は `../decisions/editorial_style_guide_v1.md` へ統合するか、本書を正本化する。

## 1. サイトの目的

Deploid は、ヒューマノイドロボットに関心を持ち始めた日本の toB 事業者が、導入判断に必要な公開情報を日本語で整理して読める入口メディアである。

単なるニュース翻訳、プレスリリース転載、研究紹介ではない。読者が知りたいのは「すごいか」ではなく、「自社で検討するなら、何を確認すべきか」「この発表は調達・PoC・安全・運用・費用のどこに効くのか」である。

したがって、記事は以下を満たす必要がある。

- 公開情報を事実として整理する
- その事実が導入判断にどう関係するかを説明する
- まだ分からないこと、確認が必要なことを曖昧にしない
- 読者が次に調べるべき問いを具体化する

## 2. 想定読者

主読者は、日本企業でヒューマノイド導入を検討する人である。技術研究者ではなく、現場・予算・稟議・安全・運用に責任を持つ人を想定する。

具体的には以下。

- 製造業の設備・生産技術・DX担当
- 物流、倉庫、EC、施設管理の現場改善担当
- 小売、接客、介護、空港、ビルメンなどの新規事業・業務企画担当
- メーカー、代理店、SIer、投資家など、市場動向を見たい関係者

読者の前提知識は高く見積もらない。専門用語は使ってよいが、導入判断に関係する形で説明する。技術の詳細そのものを読ませるのではなく、技術が実務上の制約や選定条件にどう変換されるかを書く。

## 3. 読者のニーズ

読者のニーズは、ニュース消費ではなく判断材料の収集である。

重要な問いは以下。

- どのメーカー・機体が、どの現場で、どの段階まで進んでいるか
- 実証、商用導入、量産、予約販売、研究発表のどれなのか
- 何台、どこで、誰が、何の作業をしているのか
- 安全、保守、遠隔支援、データ、現場改修は誰が担うのか
- 日本企業が今すぐ見るべき点と、まだ様子見でよい点は何か
- 自社でPoCをやるなら、どの条件を事前に確認すべきか

記事は、この問いに答えられる題材だけを扱う。答えが出せない題材は、記事ではなくメモや調査候補に留める。

## 4. 扱うべき題材

優先度が高い題材は、導入判断に直結するもの。

### A. 実導入・商用契約

最優先。工場、倉庫、店舗、施設、空港、介護現場などで、実際にロボットが使われた、契約された、台数や作業内容が出たもの。

見るべき点:

- 導入企業、現場、作業内容
- 台数、期間、契約形態
- 実証か商用か
- 成功指標、稼働実績、制約
- 日本の類似現場に置き換えた時の確認事項

### B. 製品・価格・販売条件

機体、価格、販売地域、RaaS、サブスク、予約開始、納期、サポート体制に関する発表。

見るべき点:

- 買えるのか、予約だけか、研究機か
- 日本から調達できるのか
- 価格に何が含まれるのか
- 保守、遠隔支援、交換部品、ソフトウェア更新はどうなるのか

### C. 量産・供給体制・資金調達

メーカーの量産能力、工場、資金調達、IPO、主要顧客、パートナーシップ。

見るべき点:

- 供給継続性
- サポート継続リスク
- 価格下落の可能性
- 日本市場に入ってくる可能性
- ベンダー選定時の信用材料

### D. 安全・規制・認証・標準

人と同じ空間で動かすための安全設計、第三者認証、政策、規制、標準化。

見るべき点:

- 何が決まったのか、まだ準備段階なのか
- どの標準・認証・規制に関係するのか
- 導入企業側の責任範囲はどこか
- PoC前に確認すべき安全条件は何か

### E. 大手企業との提携・導入準備

自動車、物流、EC、空港、製造大手などが関わる発表。

見るべき点:

- 実作業に近いのか、広報デモなのか
- 既存ライン・施設にどう入れる想定か
- その企業が何を検証しているのか
- 他社が参照できる条件はあるか

## 5. 扱いを下げる題材

以下は原則として優先度を下げる。書く場合も、単独記事ではなく週次メモ、調査メモ、背景記事の一部に留める。

- arXiv などの論文単体
- 研究室デモだけで、実運用や導入条件に接続できないもの
- ロボットが踊る、走る、殴る、話題化するだけの動画
- 「世界初」「業界初」だけが売りで、条件が薄い発表
- 出典が1本しかなく、公式確認も取れない噂
- 技術用語を説明して終わるだけの記事

論文や研究は完全に禁止ではない。ただし記事化する条件は厳しくする。商用導入、安全規制、製品仕様、PoC設計に直接つながる場合だけ扱う。研究そのものの紹介は Deploid の中心ではない。

## 6. 記事の切り口

記事の切り口は「何が起きたか」ではなく、「導入判断のどの変数が動いたか」で決める。

主な変数:

- 調達可能性: 買えるのか、借りられるのか、まだ研究段階か
- 価格/TCO: 本体価格、運用費、保守、遠隔支援、周辺設備
- 安全: 人協働、停止条件、認証、事故時責任
- 運用: 誰が教えるか、誰が保守するか、遠隔支援が必要か
- 現場適合: 床、段差、通路幅、照明、通信、作業対象のばらつき
- 供給体制: 量産、代理店、部品、サポート、会社継続性
- データ: 現場映像、操作ログ、学習データの所有権と再利用

同じニュースでも、どの変数が動いたかで記事の価値が変わる。変数が1つも動かない題材は、記事にしない。

## 7. 書き方の基本

記事本文は、読者の判断を前に進めるために書く。見た目だけの分析項目や、AIっぽい定型見出しは使わない。

### 使わない見出し

- Deploid読者への示唆
- なぜ重要か
- 導入企業が見るべきチェック項目
- まとめ
- 今後の注目点
- この記事のポイント

これらは編集者の都合であって、読者の読みたい見出しではない。見出しは内容そのものを言う。

悪い例:

```markdown
## Deploid読者への示唆
```

良い例:

```markdown
## 遠隔支援込みの運用費を先に確認する必要がある
```

### 文体

- だ・である調
- 煽らない
- 褒めない
- 未来を断言しない
- 事実、条件、未確認点を分ける
- 「可能性がある」を連発しない
- 「重要だ」で段落を閉じない。何を確認するかまで書く

### 長さ

短すぎる記事は出さない。最低でも、本文だけで以下を満たす。

- news-brief: 1,200〜1,800字
- analysis / tech-update / deployment-report: 2,000〜3,500字
- market-analysis / policy-update: 2,500〜4,000字

短くなる題材は、記事化するほどの材料がない可能性が高い。無理に記事にせず、複数ニュースをまとめるか、調査待ちにする。

## 8. 構成

基本構成は次の4段階。

1. 何が起きたか
2. どこまで事実として確認できるか
3. 導入判断のどの変数に関係するか
4. 読者が次に確認すべきこと

ただし、見出しにこのラベルをそのまま使わない。見出しは具体化する。

例:

```markdown
## Robot.comは二足歩行ではなく職場タスクへの近さを売る

## 公式サイトで確認できること、報道でしか確認できないこと

## 初期自律率より遠隔支援の設計が調達条件になる

## PoC前に確認すべき契約条件
```

## 9. 情報源の扱い

記事化の最低条件:

- 公式発表または信頼できる報道が1本以上ある
- 公開記事として出す場合は、原則2本以上の出典を持つ
- 数字、日付、社名、台数、顧客名は出典にあるものだけを書く
- 公式未確認の報道は「報じた」と明記する
- 画像は権利確認できない限り使わない

出典が弱い場合は、記事の断定を弱めるのではなく、記事化しない判断を優先する。

## 10. 記事化判定

記事化前に以下を確認する。

- この題材は導入判断のどの変数を動かすか
- 読者はこの記事を読んだ後、何を判断できるようになるか
- 公式または信頼できる出典があるか
- 既存記事と重複していないか
- 本文2,000字以上で実のある説明ができるか
- 画像なしでも一覧/詳細で耐える題材か
- heroに出すだけの鮮度、重要度、見た目の条件があるか

1つでも弱い場合は、記事化ではなく候補メモにする。

## 11. hero掲載基準

heroは「最新」だけで選ばない。以下を満たすものだけ載せる。

- 数日以内のニュースまたは公式発表
- Deploidの読者にとって導入判断上の意味が明確
- 既存記事より優先して見せる理由がある
- 表示可能な画像がある、または画像なしでも専用UIで破綻しない
- タイトルがhero表示に耐える長さになっている

画像なし記事をheroに置く場合は、タイトルを短くし、No Image表示に依存しないデザインが必要である。

## 12. レビュー観点

この草案で確認したい点:

- Deploidの記事は「ニュース」寄りか「導入判断メディア」寄りか
- 論文・研究をどこまで扱うか
- 最低本文量をどこに置くか
- hero掲載に画像必須条件を置くか
- `whyItMatters` や `keyTakeaways` を表に出すか、データ内部の編集補助に留めるか
- 「読者が次に確認すべきこと」を記事本文でどこまで具体化するか
