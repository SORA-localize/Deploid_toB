# AI実装ワークフロー用プロンプト集

AIに計画、実装、リファクタリング、レビュー、検証を任せるときのための共通プロンプト集です。

この文書の目的は、AIにその場しのぎの変更をさせず、既存プロジェクトの設計、責務、状態管理、検証方法に沿って継続的に開発させることです。
Webサイト構築だけでなく、一般的なソフトウェア開発、UI実装、データ設計、運用改善、AIによるバイブコーディング全般で使える粒度にしています。

基本方針は以下です。

1. 既存コードを調査して、事実ベースで計画する
2. 影響範囲とリスクを先に明確にする
3. 小さく実装し、既存の設計に合わせる
4. 自己監査と検証を挟む
5. 変更内容、残リスク、手動確認項目を明示する

---

## 0. 最重要原則

すべての作業で、以下を確認してください。

- Existing Code First: 既存の構造、命名、責務分離、型、状態管理、UI部品、検証コマンドを先に調べる
- Minimal Correct Change: 目的を満たす最小変更にする。不要な大規模リファクタや美化を混ぜない
- Single Source of Truth: データ、定数、ラベル、URLパラメータ、仕様値、状態の正本を散らばらせない
- Separation of Concerns: UI、データ取得、状態管理、URL操作、整形処理、バリデーション、副作用を不必要に混ぜない
- DRY: 同じ構造、同じロジック、同じ見た目、同じ定数を重複させない
- KISS: 抽象化しすぎない。実際の重複や複雑性を減らす抽象化だけを追加する
- Type Safety: 自由入力の文字列、曖昧な型、暗黙のデータ形状に依存しない
- Behavior Preservation: 明示された目的以外の既存挙動を変えない。変わる場合は理由と影響を説明する
- State Integrity: URL state、local state、server state、cache、永続化データの責務と同期関係を明確にする
- Operational Safety: API key、token、フォームID、顧客データ、外部サービスID、環境変数を不用意に直書きしない
- Accessibility: UIでは意味のあるHTML、label、aria、キーボード操作、フォーカス状態を確認する
- Responsive / Layout Safety: 固定幅、固定高、内部スクロール、z-index、overflow、モバイル表示の破綻を確認する
- Error / Empty / Loading States: 正常系だけでなく、空状態、失敗、読み込み中、データ欠損、境界値を扱う
- Performance Awareness: 不要な再計算、重いレンダー、過剰な通信、巨大な依存追加を避ける
- Testability: 変更した責務に合う検証方法を選び、実行できない場合は代替確認を示す

## 0.5. AI実装で特に避けること

AIは局所最適な修正を作りやすいため、以下を禁止または強く警戒してください。

- 既存コードを調べずに、新しい構造や命名を作る
- 似た実装があるのに、ページや機能ごとに独自実装を増やす
- バグを直すために、意図された機能、アニメーション、UI、データを消す
- 型エラーやビルドエラーを隠すために、`any`、不要なoptional、空配列、空文字、条件分岐でごまかす
- 仕様値、URLパラメータ名、状態名、ラベル、上限値を複数箇所に直書きする
- 一時的な見た目調整を、仕様やデータ設計に混ぜる
- UIコンポーネントにデータ取得、検索条件生成、URL更新、ビジネスルールを詰め込む
- 影響範囲が広い共通部品を、局所的な都合だけで変更する
- 不要なライブラリを追加する。追加する場合は既存依存、バンドル、保守性、代替案を確認する
- 検証していないのに「問題なし」と言う
- ユーザーの既存変更や未コミット変更を勝手に戻す

## 0.6. 設計判断で確認すること

計画、実装、リファクタリングでは、以下の問いを使って判断してください。

- これは既存のどの責務、モジュール、レイヤーに属するか
- 既存に似た実装、型、helper、hook、component、constant はあるか
- 新しく作る場合、なぜ再利用では足りないのか
- この変更で正本になるデータや状態はどこか
- 変更後に同じことを別ページ、別機能、別データでやる場合、無理なく拡張できるか
- 今回は共通化すべきか、まだ局所実装に留めるべきか
- ユーザーに見える挙動、URL、データ、アクセシビリティ、パフォーマンスに副作用はあるか
- 失敗、空、欠損、境界値、権限なし、ネットワーク不調ではどう動くか
- 将来の運用、表示枠追加、設定追加、データ追加、仕様変更で破綻しにくいか
- 検証はコマンド、型、テスト、手動確認、画面確認のどれで担保するか

## 1. 計画作成フェーズ

実装前に使うプロンプトです。

```text
まず実装せず、既存コードを調査して計画だけ作成してください。

目的:
{ここにやりたいことを書く}

必ず確認すること:
- 既存のディレクトリ構成、命名規則、型、定数、helper、hooks、components、状態管理、検証コマンド
- 似た既存実装があるか。ある場合は再利用するか、再利用しない理由
- 今回の変更が属する責務、レイヤー、モジュール境界
- 正本になるデータ、状態、URLパラメータ、設定値、ラベル、仕様値
- 既存挙動を維持する部分と、意図的に変える部分
- 影響範囲、依存関係、副作用、リスク
- 空状態、エラー、読み込み中、データ欠損、境界値の扱い
- UIがある場合はアクセシビリティ、レスポンシブ、フォーカス、スクロール、overlay、z-index
- 実装後に実行できる検証コマンドと手動確認項目

計画に含めるもの:
- 調査したファイル
- 再利用する既存コード
- 新規作成するコード
- 変更するファイル
- 変更しないファイル
- 実装手順
- 影響範囲
- リスクと軽減策
- 検証コマンド
- 手動確認チェックリスト

禁止:
- 既存コードを調べずに新規実装しない
- 大きなリファクタを勝手に混ぜない
- 既存の型、定数、ラベル、URLパラメータ、状態名を重複定義しない
- 実行できない検証を実行済みのように書かない

設計判断は断言だけで済ませず、理由、代替案、残るリスク、確認方法をセットで書いてください。
```

## 1.5. 計画の監査フェーズ

計画が出たあと、すぐ実装せず、計画自体を疑わせるためのプロンプトです。

```text
この計画に問題が10個あると仮定して、徹底的に監査してください。

特に以下を確認してください。

- 目的に対して変更範囲が広すぎないか、狭すぎないか
- 既存構造、命名、責務、型、状態管理と矛盾していないか
- DRY / KISS / Single Source of Truth / Separation of Concerns に反していないか
- データ、状態、定数、ラベル、URLパラメータが散らばらないか
- UI、データ取得、状態管理、URL操作、整形処理、副作用が混ざらないか
- 型安全性が弱くならないか
- 既存挙動を壊す可能性がある箇所はどこか
- エラー、空状態、読み込み中、データ欠損、境界値の扱いが不足していないか
- UIがある場合、アクセシビリティ、レスポンシブ、スクロール、overlay、z-indexに問題がないか
- セキュリティ、秘密情報、外部サービス、運用上の安全性に問題がないか
- 検証方法が変更内容に対して不足していないか

出力形式:
1. 問題点
2. 重大度
3. なぜ問題か
4. 修正方針
5. 修正後の計画への反映方法
```

監査後、計画を修正させます。

```text
上記の監査結果を反映して、実装計画を修正してください。

修正後の計画には以下を必ず含めてください。

- 反映した問題点
- 反映しなかった問題点と理由
- 最終的な変更ファイル一覧
- 実装順序
- 影響範囲
- 検証コマンド
- 手動確認項目
- 残るリスク

まだ実装は開始しないでください。
```

## 2. 実装フェーズ

計画に納得したあとに使うプロンプトです。

```text
修正済みの計画に沿って実装してください。

実装ルール:
- 計画に書かれていない大きな変更はしない
- 既存コードの命名規則、ディレクトリ構成、import、型定義、UI部品、状態管理、スタイリング方針に合わせる
- 既存の型、定数、ラベル、helper、hooks、components がある場合は再利用する
- 新しい定数、ラベル、URLパラメータ、仕様値を追加する場合は、正本になる場所に置く
- UI表示、データ取得、整形処理、状態管理、URL操作、副作用を不必要に混ぜない
- バグ修正では、意図された既存機能を消さず、原因を特定して最小修正する
- リファクタリングでは、外部挙動を変える変更と構造改善を混ぜない
- 共通部品を変更する場合は、利用箇所と副作用を確認する
- ハードコードする場合は、一時的な見た目調整なのか、仕様値なのかを区別する
- 既存機能を壊す可能性がある変更は、最小差分で行う

実装中に計画と違う判断が必要になった場合:
- 勝手に大きく方針変更しない
- なぜ変更が必要かを説明する
- 代替案を示す
- 最小変更で進める
```

## 2.5. 実装中の自己確認

実装がある程度進んだところで使うプロンプトです。

```text
ここまでの実装を一度確認してください。

確認項目:
- 計画と違う実装になっていないか
- 既存コードの書き方、命名、責務、型、状態管理と整合しているか
- 重複コードや重複定数が増えていないか
- 不要な抽象化をしていないか
- データ、ラベル、仕様値、URLパラメータ、状態名が散らばっていないか
- UI、データ取得、整形処理、状態管理、URL操作、副作用が混ざりすぎていないか
- 既存挙動を意図せず変えていないか
- 型が弱くなっていないか
- エラー、空状態、読み込み中、データ欠損、境界値に対応しているか
- UIがある場合、アクセシビリティ、レスポンシブ、フォーカス、スクロール、overlay、z-indexに問題がないか
- 既存機能に副作用が出そうな変更がないか

問題があれば、重大度順に並べ、最小修正案を出してください。
```

## 2.8. リファクタリング専用フェーズ

リファクタリングを依頼するときに使うプロンプトです。

```text
まず実装せず、リファクタリング計画だけ作成してください。

目的:
{整理したい問題、重複、責務混在、保守性の課題を書く}

必ず確認すること:
- 現在の問題が、重複、責務混在、型の弱さ、状態管理の曖昧さ、命名不一致、データ設計、UI構造、依存方向のどれに当たるか
- 外部挙動を変える必要があるか。ない場合は挙動維持を前提にする
- 既存の公開API、props、URL、データ形式、保存済みデータ、利用箇所への影響
- どの単位で分割、移動、抽出、統合するか
- 抽象化の導入理由。実際の重複や複雑性が減るか
- 一度にやるべき変更と、別PR/別コミットに分けるべき変更
- リファクタ前後で同じ挙動を確認する方法

禁止:
- 見た目変更、仕様変更、機能追加を無断で混ぜない
- 共通化のためだけに読みにくい汎用部品を作らない
- 型やテストを弱めて移動だけ成功させない
- 既存の利用箇所を確認せずに共通部品を変えない

出力には、変更前後の責務分担、移動するコード、残すコード、検証方法、残るリスクを含めてください。
```

## 3. レビュー・検証フェーズ

実装後に必ず使うプロンプトです。

```text
実装後レビューをしてください。

確認項目:
- 目的に対して変更範囲が適切か
- 既存の設計、命名、責務分離、型、状態管理、UI部品と整合しているか
- DRY / KISS / Single Source of Truth / Separation of Concerns / Type Safety に違反していないか
- 既存挙動への影響範囲はどこか
- ハードコード、重複、不要な抽象化、責務混在がないか
- エラー、空状態、読み込み中、データ欠損、境界値の扱いが十分か
- UIがある場合、アクセシビリティ、レスポンシブ、フォーカス、スクロール、overlay、z-indexに問題がないか
- セキュリティ、秘密情報、外部サービス、運用上の安全性に問題がないか
- 型エラー、lintエラー、buildエラー、testエラーがないか
- 動作確認すべき正常系、異常系、境界値は何か

以下の形式で出してください。

1. 実行した検証コマンドと結果
2. 変更ファイルごとの要約
3. 発見した問題
4. 修正した問題
5. 残っているリスク
6. 手動確認チェックリスト
```

## 3.5. 疎通確認・回帰確認

画面や機能が実際に既存アプリと噛み合っているかを確認するためのプロンプトです。

```text
実装後、既存アプリとの疎通確認をしてください。

確認項目:
- 対象画面または対象機能が正常に使えるか
- 主要操作がエラーなく動くか
- 状態、URL、保存データ、キャッシュ、リロード後の挙動が期待通りか
- ブラウザの戻る/進む、再読み込み、直接URLアクセスで破綻しないか
- 空状態、データ欠損、上限到達、権限なし、失敗時の表示が破綻しないか
- モバイル幅、タブレット幅、PC幅で表示が破綻しないか
- 横スクロール、内部スクロール、sticky、overlay、dropdown、modal がある場合、操作性に問題がないか
- button / input / link / form がキーボード操作できるか
- aria-label、label、role、aria-selected、aria-expanded などが必要な箇所にあるか
- コンソールエラー、ネットワークエラー、サーバーログの異常がないか
- 既存ページ、既存コンポーネント、既存APIに副作用が出ていないか
- 型、lint、build、test が通るか

実行できない確認がある場合は、実行できなかった理由と、代替確認方法を明記してください。
```

## 4. 検証コマンドの特定

プロジェクトによってコマンド名は違います。まず `package.json`、README、CI設定、Makefile、GitHub Actions などを確認させてください。

```text
このプロジェクトで使える検証コマンドを特定してください。

確認対象:
- package.json / pnpm scripts / yarn scripts
- README / docs
- Makefile / task runner
- CI設定
- 型チェック、lint、format、unit test、integration test、e2e test、build、data validation

存在しないコマンドを勝手に実行しないでください。
存在しない場合は、その旨を明記してください。
```

一般的な候補:

```bash
npm run lint
npm run typecheck
npm run build
npm run test
npm run test:unit
npm run test:e2e
```

実際には、そのプロジェクトに存在する scripts だけを使ってください。

最後に、Pull Request レビュー風に確認させるプロンプトです。

```text
今回の変更をPull Requestレビューのつもりで確認してください。

レビュー観点:
- 目的に対して変更範囲が適切か
- 既存実装の流儀に合っているか
- 将来の変更に弱い設計になっていないか
- 不要なハードコードがないか
- 重複や過剰抽象化がないか
- 型安全性が落ちていないか
- 状態管理、URL、データ構造、永続化、キャッシュに矛盾がないか
- エラー、空状態、読み込み中、データ欠損、境界値の扱いが十分か
- UIがある場合、アクセシビリティ、レスポンシブ、操作性に問題がないか
- セキュリティ、秘密情報、外部サービス、運用上の安全性に問題がないか
- 既存機能に副作用がないか
- 検証コマンドと手動確認が変更内容に対して十分か

出力形式:
- Approve / Request changes / Comment のどれか
- 理由
- 修正必須の項目
- 任意改善の項目
- 次に確認すべきこと
```

## 6. 使い方の基本手順

おすすめの流れは以下です。

```text
1. 計画作成フェーズのプロンプトを投げる
2. 計画の監査フェーズのプロンプトを投げる
3. 監査結果を反映して計画修正させる
4. 実装フェーズのプロンプトを投げる
5. 実装中の自己確認を挟む
6. レビュー・検証フェーズのプロンプトを投げる
7. 疎通確認・回帰確認をさせる
8. PRレビュー風の最終確認をさせる
```

実装が小さい場合は、1、3、6だけでも十分です。
リファクタリング、共通部品変更、状態管理変更、データ構造変更、認証、決済、顧客データ、外部サービス連携が絡む場合は、全段階を通してください。
